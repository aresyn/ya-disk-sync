# Модель синхронизации

`ya-disk-sync` делает одностороннее зеркало:

```text
локальный диск -> Яндекс Диск
```

Локальный каталог считается источником истины. Если файл есть локально, он должен быть в управляемом облачном каталоге. Если файл есть в облаке, но исчез локально, он удаляется из облака.

## Почему это не обычная облачная папка

Проект не пытается разруливать двусторонние конфликты. Он подходит для ночного зеркала рабочих каталогов, где важно иметь актуальную копию в Яндекс Диске и не начинать огромный прогон с нуля после сбоя.

## SQLite-state

State хранится в SQLite и содержит:

- историю запусков;
- список roots;
- состояние файлов и каталогов;
- кэш удалённого inventory;
- skipped/failed items;
- журнал операций;
- migration map;
- lock одного активного запуска.

Благодаря этому повторный запуск использует накопленный прогресс: уже принятые файлы не загружаются повторно, а удалённый inventory можно брать из кэша до истечения TTL.

## Сравнение файлов

- Малые файлы до `full_hash_max_bytes` проверяются по хэшу.
- Средние файлы сравниваются по имени, размеру и времени изменения, когда это безопасно.
- Крупные файлы от `large_file_name_size_only_min_bytes` сравниваются по имени и размеру.
- Файлы больше `max_file_size_bytes` пропускаются и попадают в skipped items.

SQLite-файлы `.sqlite`, `.sqlite3`, `.db` читаются через SQLite Online Backup API: сначала создаётся консистентная staging-копия, затем fingerprint считается по ней. WAL/SHM исключаются отдельно.

## Удаления

Удаления выполняются в конце, после create/upload/update. Это снижает риск потерять удалённую копию, если upload-фаза упала.

Если при удалении Яндекс Диск возвращает `NotFound`, операция считается идемпотентно успешной: нужного лишнего файла уже нет. Некорректные phantom paths вида `<root>/disk:` не попадают в план удаления.

## Удалённые подкаталоги

Если внутри root локально исчез целый подкаталог, sync не должен пытаться обработать тысячи бывших файлов по одному. Он создаёт tombstone подкаталога и планирует один delete subtree в облаке. Root целиком защищён отдельно: если отсутствует сам local root, удаление remote root запрещается.

## Надёжность

Transient-сетевые ошибки ретраятся. Если remote listing по root не удалось получить полностью, delete phase для этого root не запускается: неполный snapshot не считается безопасным основанием для удаления.
